# Trace Contract

`Trace` records one MemoryNexus interaction or execution with enough runtime,
provenance, and feedback data to support local-first evaluation and future
learning loops.

Trace is not a user-facing note. It is an internal evidence object that connects
runtime behavior to memory evolution and feedback quality.

## Goals

- Record interaction provenance for Lens Run, Review Report, MCP tool calls,
  FeedbackLoop updates, and practice sessions.
- Track runtime metrics such as latency, token usage, estimated cost, and
  local/cloud/deterministic execution mode.
- Link generated objects back to the interaction that produced them.
- Support namespace-specific feedback effectiveness analysis.
- Keep `CognitiveSpace` as the ownership and permission boundary.

## Non-Goals

- Do not build a local inference engine.
- Do not implement OpenJarvis-style model optimization or fine-tuning.
- Do not expose Trace as a primary user-facing UI label.
- Do not replace Memory, LensRun, ReviewReport, or FeedbackLoop.
- Do not store secrets, raw credentials, API keys, JWTs, or provider tokens.

## Fields

| Field | Required | Notes |
| --- | --- | --- |
| `id` | yes | Stable UUID. |
| `space_id` | yes | The owning `CognitiveSpace`; permission checks remain Space-based. |
| `namespace_id` | no | Domain partition when the interaction belongs to a namespace. |
| `source_type` | yes | Origin surface: `http`, `cli`, `mcp`, `ui`, `background`, `test_fixture`. |
| `task_type` | yes | Semantic task: `chat`, `search`, `lens_run`, `review`, `practice`, `feedback`, `planning`, `install`, `profile`, `routing`. |
| `mode` | yes | Observe depth: `fast`, `focused`, `deep`, or `none` when not applicable. |
| `runtime` | yes | Execution class: `local`, `cloud`, `hybrid`, `deterministic`, or `unknown`. |
| `input_summary` | no | Redacted or summarized input. Avoid raw secrets and oversized payloads. |
| `output_summary` | no | Redacted or summarized output. Avoid storing provider-sensitive raw output when not needed. |
| `started_at` | yes | Timestamp when the interaction began. |
| `completed_at` | no | Timestamp when the interaction finished. |
| `latency_ms` | no | Completed duration; should be present for completed traces. |
| `status` | yes | `started`, `completed`, `failed`, `cancelled`, or `skipped`. |
| `model_provider` | no | Example: `openai`, `openrouter`, `local`, `deterministic`. |
| `model_name` | no | Provider model or local runtime name. |
| `token_usage` | no | Input/output/total tokens when known. |
| `estimated_cost_usd` | no | Best-effort estimated provider cost. |
| `local_processing_ratio` | no | 0.0 to 1.0 estimate of local vs remote processing. |
| `related_memory_ids` | no | Existing memories read or cited. |
| `generated_memory_ids` | no | Memories created by this interaction. |
| `generated_lens_run_ids` | no | Lens Runs generated by this interaction. |
| `generated_review_report_ids` | no | Review Reports generated by this interaction. |
| `generated_feedback_loop_ids` | no | FeedbackLoops created or updated by this interaction. |
| `generated_reflection_ids` | no | Reflections generated by this interaction when that model exists. |
| `user_feedback` | no | Optional rating/usefulness/note. |
| `error` | no | Redacted error class/message for failed traces. |
| `metadata` | no | Small structured extension point. |

## Relationships

Trace links objects; it does not own them.

```text
CognitiveSpace
  └── Trace
        ├── Namespace?
        ├── Memory*
        ├── LensRun*
        ├── ReviewReport*
        ├── FeedbackLoop*
        └── Reflection*
```

All linked objects must belong to the same `CognitiveSpace`. If a Trace has
`namespace_id`, linked namespace-scoped objects must either share that namespace
or explicitly record why they are cross-namespace evidence.

## Lifecycle

Minimum lifecycle:

```text
start trace
→ run interaction
→ attach generated object IDs
→ record metrics
→ complete or fail trace
→ optionally collect user feedback later
```

The first implementation may create only completed traces if that is simpler.
Long-running background work can later use `started` and `completed` states.

## Privacy And Retention

Trace may contain sensitive input/output summaries. Implementations must:

- Redact secrets, tokens, passwords, credentials, and private keys.
- Prefer summaries or hashes over raw large payloads.
- Avoid storing full third-party provider responses unless needed for provenance.
- Keep Trace access under `CognitiveSpace` membership.
- Document any future retention or deletion policy before enabling broad trace
  capture.

## First Capture Surfaces

Recommended order:

1. Lens Run
   - `task_type = lens_run`
   - `mode = focused` or `deep`
   - link matched Memory IDs and generated LensRun ID
2. MCP tool call
   - `source_type = mcp`
   - `task_type` based on tool category
   - useful for agent install and practice flow debugging
3. FeedbackLoop / STEM practice session
   - `task_type = practice` or `feedback`
   - link generated Memory and FeedbackLoop IDs
4. Weekly learning review
   - `task_type = review`
   - link source FeedbackLoop / Memory IDs and generated report ID

## Metrics

Initial metrics:

- `latency_ms`
- `mode`
- `runtime`
- `model_provider`
- `model_name`
- `token_usage`
- `estimated_cost_usd`
- `local_processing_ratio`

Later metrics:

- energy estimate
- FLOPs estimate
- cache hit rate
- trace reuse count
- improvement per practice cycle

Energy and FLOPs should remain future work until there is a reliable local
measurement strategy.

## STEM Learning Feedback Example

```text
Trace:
  source_type: mcp
  task_type: feedback
  mode: focused
  runtime: deterministic
  namespace: learning.stem
  input_summary: parent recorded learner answer and feedback for a fraction word problem
  output_summary: practice session updated and memory snapshot captured
  generated_feedback_loop_ids: [...]
  generated_memory_ids: [...]
  latency_ms: 42
  user_feedback:
    useful: true
```

The weekly review can later compare traces across attempts:

```text
feedback trace
→ next attempt trace
→ improvement signal
→ suggested next practice
```

## Validation Questions

Before implementing Trace persistence, answer these in the issue or PR:

- Which surfaces create Trace records first?
- Are traces written synchronously or best-effort after the main operation?
- What input/output fields are redacted?
- Which linked objects must be same-Space validated?
- How does the implementation avoid failing the user operation when trace
  capture fails?
- Which metrics are exact, estimated, or unavailable?
